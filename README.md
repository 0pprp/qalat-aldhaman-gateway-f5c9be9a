# QalatAldhaman.Store.Admin

تطبيق سطح مكتب (WPF, .NET 8, Windows) للوحة تحكم إدارة متجر قلعة الضمان — بديل عن استخدام Swagger يدوياً لإدارة الفئات، المنتجات، الطلبات، والآراء عبر واجهة رسومية مباشرة.

مشروع **منفصل تماماً** عن الباك اند (`QalatAldhaman.Store.Api`)، الفرونت اند العام (`qalat-aldhaman-gateway-main`)، ونظام CRM. يتصل بالباك اند عبر HTTP فقط.

تسجيل الدخول (JWT) جاهز وشغّال فعلياً. تبويب "الفئات" جاهز بصلاحية كاملة (عرض/إضافة/تعديل/حذف). باقي شاشات الإدارة (المنتجات/الطلبات/الآراء) لا تزال بدون وظائف فعلية — تُبنى بالبرومبتات القادمة.

## المتطلبات

- [.NET 8 SDK](https://dotnet.microsoft.com/download/dotnet/8.0)
- Windows (WPF يعمل فقط على Windows)
- **مشروع الباك اند (`QalatAldhaman.Store.Api`) يجب أن يعمل بالتوازي** أثناء التطوير — راجع الـ README الخاص به لتشغيله محلياً (`dotnet run`، افتراضياً على `http://localhost:5228`).

## هيكلة المشروع

```
Views/             # LoginView.xaml, CategoriesView.xaml, CategoryEditDialog.xaml + شاشات أخرى تُضاف لاحقاً
ViewModels/        # MVVM ViewModels (CommunityToolkit.Mvvm)
Models/Auth/       # LoginRequest, LoginResponse
Models/Categories/ # CategoryDto, CategoryUpsertRequest
Models/Uploads/    # UploadResponse
Models/            # ApiErrorResponse — نماذج DTOs أخرى تُضاف مع كل شاشة إدارة لاحقاً
Services/          # ApiClient (HttpClient + توكن)، SessionService (الجلسة بالذاكرة)، JsonDefaults، ApiMessageReader
```

## ضبط عنوان الباك اند

عنوان الباك اند موجود بـ `appsettings.json`:

```json
{
  "ApiSettings": {
    "BaseUrl": "http://localhost:5228"
  }
}
```

هذا هو عنوان التطوير المحلي الافتراضي (نفس منفذ `dotnet run` بمشروع الباك اند). **عند النشر الفعلي لاحقاً**، غيّر هذه القيمة لعنوان السيرفر الحقيقي (مثلاً `https://api.qalataldhaman.com`) — إما بتعديل `appsettings.json` مباشرة قبل النشر، أو عبر متغير بيئة/ملف إعدادات منفصل حسب طريقة التوزيع النهائية (لم تُحدَّد بعد بهذه المرحلة).

## التشغيل محلياً

1. شغّل الباك اند أولاً بمشروع `QalatAldhaman.Store.Api` (`dotnet run`).
2. بمجلد هذا المشروع:

   ```bash
   dotnet run
   ```

تفتح شاشة تسجيل دخول فارغة أولاً (لا تعبئة تلقائية، لا "تذكرني"). بعد إدخال بيانات أدمن حقيقية صحيحة، تنتقل لـ "لوحة تحكم قلعة الضمان" بقائمة التنقل الجانبية (الفئات، المنتجات، الطلبات، الآراء، الإحصائيات) — الضغط على أي عنصر يعرض حالياً فقط عنوان الصفحة (المحتوى الفعلي يُبنى بالبرومبتات القادمة).

## تسجيل الدخول والجلسة

- **لا تخزين دائم إطلاقاً**: التوكن وبيانات المستخدم تعيش فقط بالذاكرة (`Services/SessionService.cs`, Singleton). إغلاق التطبيق يفقد الجلسة نهائياً — إعادة الفتح تتطلب تسجيل دخول من الصفر دائماً. لا "تذكرني"، لا Credential Manager، لا ملف إعدادات محلي يحوي التوكن.
- **حماية Brute Force بالواجهة**: بعد 5 محاولات فاشلة متتالية، يتعطل زر الدخول 30 ثانية مع رسالة واضحة (بالإضافة لأي حماية حالية/مستقبلية بالباك اند).
- **ربط تلقائي بالتوكن**: `Services/ApiClient.cs` يقرأ التوكن من `SessionService` ويُرفقه كـ `Authorization: Bearer ...` بكل طلب. أي رد `401` على طلب كان يحمل توكناً بالفعل (وليس محاولة تسجيل دخول عادية) يُفرِّغ الجلسة فوراً ويُعيد المستخدم لشاشة الدخول برسالة "انتهت الجلسة".
- **قفل تلقائي بعد خمول**: 20 دقيقة بدون أي تفاعل (حركة ماوس/لوحة مفاتيح) بـ `MainWindow` تُنهي الجلسة تلقائياً بنفس آلية انتهاء الجلسة أعلاه. نُفِّذ مباشرة بهذا البرومبت (لم يُؤجَّل) عبر `DispatcherTimer` بسيط بـ `MainWindow.xaml.cs` — لا يحتاج مكتبة إضافية.
- **تسجيل خروج**: زر بأسفل الشريط الجانبي يمسح الجلسة فوراً ويعيد لشاشة الدخول.

## إدارة الفئات

تبويب "الفئات" (`Views/CategoriesView.xaml` + `CategoriesViewModel`) مربوط مباشرة بـ `/api/admin/categories`:

- **العرض**: عند فتح التبويب، جلب تلقائي لكل الفئات (فعّالة وغير فعّالة) بجدول يعرض الاسم، ملخص طرق الدفع المسموحة، هل تشترط صاحب محل، والحالة. مؤشر تحميل + رسالة خطأ عربية مع زر إعادة محاولة عند فشل الاتصال.
- **الإضافة/التعديل**: نافذة منبثقة (`Views/CategoryEditDialog.xaml` + `CategoryEditViewModel`) بكل حقول سكيمة الفئة (الاسم، الوصف، الصورة، طرق الدفع، صاحب محل، الحدود الدنيا للفاتورة، حقل مخصص، ترتيب العرض، فعّالة). التحقق الوحيد بالواجهة: الاسم غير فارغ — أي تحقق آخر (كحدود الفاتورة) يُترك للباك اند وتُعرض رسالته عند الفشل.
- **رفع الصورة**: زر "اختيار صورة" يفتح `OpenFileDialog` محلي، يرفع الملف فوراً عبر `POST /api/uploads?folder=categories`، ويعرض معاينة مصغّرة بعد النجاح. `ApiClient.ResolveMediaUrl` يحوّل المسار النسبي المُرجَع من الباك اند لعنوان كامل قابل للعرض بعنصر `Image`.
- **الحذف**: تأكيد بسيط (`MessageBox`) قبل التنفيذ. الباك اند قد يُرجع حذفاً نهائياً (`204 No Content`) أو تعطيلاً (`200 OK` + رسالة، إن كانت الفئة مرتبطة بمنتجات) — الرسالة الفعلية المُرجعة من الباك اند تُعرض للأدمن حرفياً لتوضيح أي السلوكين حصل.
- بعد أي عملية ناجحة (إضافة/تعديل/حذف)، إعادة تحميل تلقائية للقائمة لتعكس حالة قاعدة البيانات الحقيقية فوراً.
- `Services/ApiMessageReader.cs`: قارئ مشترك لحقل `message` من أي رد (خطأ أو نجاح مع رسالة، كحالة التعطيل بدل الحذف).

## البنية التقنية

- **MVVM** عبر `CommunityToolkit.Mvvm` (`ObservableObject`, `[ObservableProperty]`, `[RelayCommand]`).
- **Dependency Injection** عبر `Microsoft.Extensions.Hosting` — مُهيّأ بـ `App.xaml.cs`، يقرأ `appsettings.json` تلقائياً ويُسجّل `SessionService` (Singleton)، `ApiClient` (Typed HttpClient)، وشاشات `LoginView`/`MainWindow` (Transient — نافذة WPF لا يمكن إعادة استخدامها بعد `Close()`، فكل ظهور جديد ينشئ نسخة جديدة).
- **`ShutdownMode="OnExplicitShutdown"`** (مضبوط بالكود بـ `App.xaml.cs`): الافتراضي `OnLastWindowClose` كان يُنهي التطبيق بالكامل لحظة إغلاق `MainWindow` أثناء تسجيل الخروج (قبل حتى ظهور شاشة الدخول الجديدة) — تحويلها لـ `OnExplicitShutdown` مع تتبّع يدوي عبر علم `_isTransitioning` يضمن التبديل بين النوافذ بسلاسة، وأن التطبيق يُغلق فعلياً فقط عند إغلاق آخر نافذة بقرار حقيقي من المستخدم (وليس أثناء انتقال داخلي).
- `Services/JsonDefaults.cs`: `System.Net.Http.Json` بدون إعدادات صريحة يستخدم مطابقة حساسة لحالة الأحرف افتراضياً، بينما الباك اند يرجع camelCase — بدون هذا الإعداد كل تحويل JSON يفشل بصمت (خصائص فارغة، بدون استثناء ظاهر).

## ملاحظة مهمة

لا يوجد بأي ملف مرفوع لـ Git أي بيانات اتصال أو أسرار حقيقية. `appsettings.json` الحالي يحوي فقط عنوان تطوير محلي عام (`localhost`)، وليس بيانات حساسة. لا يوجد أي مسار بالكود يكتب التوكن أو كلمة المرور لأي ملف على القرص — تأكدت من هذا فعلياً بفحص الملفات المُنشأة أثناء وبعد تشغيل التطبيق.
